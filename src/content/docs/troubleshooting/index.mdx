When something isn't working, start here. This page covers the fastest diagnostic checks, then guides you to the right sub-page based on the symptom you're seeing.

## Quick checks

Run through these first — they catch the majority of issues:

1. **<Tooltip term="bedrock" label="Bedrock" /> model access enabled?** — Both Titan Embeddings V2 and your LLM must be enabled in the Bedrock console for the deployment region.
2. **Correct region?** — S3 Vectors, Bedrock models, and inference profiles are not available in all regions. Confirm your `terraform.tfvars` region matches where you enabled model access.
3. **Bootstrap ingest completed?** — Check the <Tooltip term="lambda" label="Lambda" /> ingest function logs for success. An empty corpus means queries return no results.
4. **<Tooltip term="cognito" label="Cognito" /> user confirmed?** — Users must be in `CONFIRMED` status. An unconfirmed user cannot authenticate to the API.
5. **Terraform apply finished cleanly?** — Partial applies leave resources in inconsistent states. Check for errors in the apply output.

<Aside type="tip">
  Most problems fall into two buckets: missing Bedrock model access (causes{" "}
  <code>AccessDeniedException</code>) or the ingest Lambda not completing (causes empty query
  results).
</Aside>

## Checking CloudWatch logs

Each Lambda function writes to a dedicated CloudWatch log group. The naming pattern is:

```bash
/aws/lambda/<project-prefix>-ingest
/aws/lambda/<project-prefix>-query
```

To tail logs for the ingest function:

```bash
aws logs tail "/aws/lambda/$(terraform output -raw ingest_function_name)" --follow
```

To tail logs for the query function:

```bash
aws logs tail "/aws/lambda/$(terraform output -raw query_function_name)" --follow
```

<Aside type="note">
  Replace the Terraform output commands with your actual function names if you're not running from
  the Terraform workspace directory.
</Aside>

## Checking corpus status

The query Lambda exposes a <code>/status</code> endpoint through <Tooltip term="api-gateway" label="API Gateway" /> that returns the current vector count and index health.

```bash
# Get the API endpoint
API_URL="$(terraform output -raw api_endpoint)"

# Check status (no auth required for /status)
curl -s "$API_URL/status" | jq .
```

You can also verify directly that vectors exist in the <Tooltip term="s3-vectors" label="S3 Vectors" /> index:

```bash
aws s3vectors list-vectors \
  --vector-bucket-name "$(terraform output -raw vector_bucket_name)" \
  --index-name "$(terraform output -raw vector_index_name)"
```

If the vector count is zero, the ingest function hasn't run successfully. Re-trigger it:

```bash
aws lambda invoke \
  --function-name "$(terraform output -raw ingest_function_name)" \
  /tmp/ingest-out.json

cat /tmp/ingest-out.json | jq .
```

<Aside type="caution">
  The bootstrap ingest runs automatically during <code>terraform apply</code> via a{" "}
  <Tooltip term="lambda" label="Lambda" /> invocation resource. If it failed during apply, you'll
  need to invoke it manually.
</Aside>
