## What is RAG?

<Tooltip term="rag" label="Retrieval-Augmented Generation" /> (RAG) is a pattern that combines vector similarity search with LLM generation. Instead of relying solely on a model's training data, RAG retrieves relevant source documents at query time and feeds them as context to the <Tooltip term="llm" label="LLM" />.

The result: answers that are grounded in real data, cite their sources, and stay current without retraining.

## Why RAG over pure LLM?

A pure LLM generates answers from its training data alone. This works for general knowledge but falls short for domain-specific, private, or rapidly changing information.

<Aside type="tip">
RAG is particularly effective when your data changes frequently. This demo ingests AWS announcements daily — the LLM can answer questions about announcements published yesterday without any model changes.
</Aside>

## Pipeline phases

This project implements a four-phase RAG pipeline:

### 1. Ingest

The <Tooltip term="lambda" label="Ingest Lambda" /> fetches RSS feeds, strips HTML, and prepares article text. Each article becomes a source document stored in S3.

### 2. Store

Each document is converted into a 1024-dimensional <Tooltip term="embedding" label="vector embedding" /> using <Tooltip term="titan-v2" label="Titan Embeddings V2" />, then stored in <Tooltip term="s3-vectors" label="S3 Vectors" /> alongside metadata (title, URL, date, feed source).

### 3. Query

When a user asks a question, the system embeds the question into the same vector space, then searches S3 Vectors for the most similar documents using <Tooltip term="cosine" label="cosine similarity" />.

### 4. Generate

Retrieved documents are scored, ranked, and assembled into a context window. The LLM (Claude via <Tooltip term="bedrock" label="Bedrock" />) generates a structured answer grounded in the retrieved context.

## How a question becomes an answer

![Request flow — from user question through API Gateway, Lambda, S3 Vectors, and Bedrock to structured answer](/s3-vectors-rag-workload/request-flow.png)

A complete request lifecycle:

1. User types a question in the web UI
2. Browser sends the question to <Tooltip term="api-gateway" label="API Gateway" /> with a JWT token
3. API Gateway validates the token via <Tooltip term="cognito" label="Cognito" /> and routes to Query Lambda
4. Query Lambda embeds the question using Titan V2 (same model used at ingest time)
5. Query Lambda calls <Tooltip term="query-vectors" label="QueryVectors" /> to find similar documents
6. Full document bodies are loaded from S3
7. Documents are scored (80% similarity + 20% recency) and ranked
8. Top documents are assembled into a context prompt
9. Claude generates a structured answer citing specific sources
10. Response returns to the browser with answer and source links

<Aside type="note">
The same embedding model must be used for both ingest and query. If you change models, existing vectors become incompatible and the index must be rebuilt.
</Aside>
